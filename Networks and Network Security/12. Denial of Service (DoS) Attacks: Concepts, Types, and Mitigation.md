# Denial of Service (DoS)
A DoS attack is a cyberattack that targets a network or server by flooding it with excessive traffic. The main objective is to disrupt normal business operations by overwhelming the system.
### Goals:
- Crash the network device
- Prevent legitimate users from accessing services
- Disrupt normal business operations
- Cost the organization money and time

## DoS vs DDoS

| Feature           | DoS (Denial of Service)                                    | DDoS (Distributed Denial of Service)                               |
| ----------------- | ---------------------------------------------------------- | ------------------------------------------------------------------ |
| **Origin Count**  | One single system and one IP address                       | hundreds to millions of distributed sources                             |
| **Orchestration** | Directly launched by the attacker                          | Managed via a Command and Control (C2) server                      |
| **Primary Tool**  | Script or software (e.g., LOIC)                            | Botnet (network of compromised “zombie” devices)                   |
| **Detection**     | Easy; traffic pattern is obvious and comes from one source | Difficult; traffic mimics legitimate users from multiple locations |
| **Volume**        | Limited by the attacker’s local bandwidth                  | Massive; can reach several terabits per second (Tbps)              |
| **Mitigation**    | Simple (IP blocking, firewall rules)                       | Complex (Scrubbing centers, Anycast DNS, CDNs required)            |

## Why is DDoS Mitigation So Difficult?
In the cybersecurity world, we often call DDoS a "Resource Exhaustion" problem. It is difficult to fight because it exploits the very nature of how the internet is designed to work.

### 1. The "Needle in a Haystack" Problem (Legitimate-looking Traffic)
Modern DDoS attacks don't just send "junk" data; they often perform Application Layer (Layer 7) attacks.
- **_The Mechanic:_** The botnet sends standard HTTP GET or POST requests—the same requests a real human makes when clicking a link.
- **_The Challenge:_** To a server, a bot from a hijacked laptop in Tokyo looks exactly like a real customer from London. If you block all traffic to stop the attack, you essentially do the attacker’s job for them by taking your own site offline.

### 2. Massive Volumetric Capacity (The Tbps Scale)
The "pipe" connecting a company to the internet has a limit.

- **_The Mechanic:_** If a company has a 10 Gbps (Gigabits per second) fiber line, and an attacker hits them with 100 Gbps of traffic, the hardware fails before it can even look at the packets.

- **_The Challenge:_** The network becomes so saturated that security administrators cannot even log in remotely to fix the problem because no data can get in or out.

### 3. IP Spoofing and Geo-Distribution
- **_The Mechanic:_** Attackers use IP Spoofing to forge the "Source IP" address in the packet header. This makes it appear as if the attack is coming from millions of random locations.

- **_The Challenge:_** Traditional firewalls work by "Blacklisting" bad IPs. When the source is spoofed or distributed across a 500,000-node botnet, blacklisting becomes useless—it's like trying to stop a flood with a single brick.


## Common Network-Level Attacks

### 1. SYN Flood Attack (Exploiting the TCP Handshake)
This is the most common network-level attack. It manipulates the TCP Three-Way Handshake process to create "half-open" connections.
- **_How it works:_** In a normal connection, a user sends a SYN, the server responds with a SYN/ACK, and the user sends a final ACK. In this attack, the attacker sends thousands of SYN packets but never sends the final ACK.
- **_Impact:_** The server allocates memory (the SYN Queue) for each of these incomplete connections. Eventually, the server exhausts available connection slots, and legitimate users are unable to connect.
- **_Defense:_** Implementing SYN Cookies. This allows the server to avoid allocating memory until the final ACK is received and validated.

### 2. ICMP Flood (Bandwidth Jamming)
Commonly known as a Ping Flood, this attack targets network bandwidth ("internet pipe") or bandwidth rather than the server's internal processing power.
- **_How it works:_** The attacker sends an overwhelming number of ICMP Echo Request (Ping) packets to the target server. By protocol, the server must process and respond to every single ping.
- **_Impact:_** Both incoming and outgoing bandwidth become saturated with useless traffic. There is no "room" left for legitimate data to travel back and forth.
- **_Defense:_** Configuring Rate Limiting on firewalls or disabling ICMP responses for non-essential external traffic.

### 3. UDP Flood (Connectionless Exhaustion)
UDP is a connectionless protocol, meaning it doesn't require a handshake. This makes it a low-cost and easy-to-exploit vector for attackers for attackers.
- **_How it works:_** The attacker sends large volumes of UDP packets to random ports on the target server.
- **_Impact:_** For every packet received, the server checks if an application is listening on that port. When it finds none, it sends back an ICMP Destination Unreachable packet. This cycle consumes CPU cycles and bandwidth, degrading overall system performance and responsiveness.
- **_Defense:_** Filtering UDP traffic at the firewall level and blocking traffic to ports that are not explicitly required for business operations.


## Defense Strategies: The Cybersecurity Expert’s Toolkit
To survive a DDoS attack, you must shift from traditional "defense" to "traffic scrubbing."

### 1. Network Level Defense
This is your first line of defense, designed to ensure your hardware and "pipes" don't collapse under the initial weight of an attack.
- **_Bandwidth Over-provisioning:_** Maintain more bandwidth than your daily peak usage requires. This acts as a "buffer" to absorb small traffic spikes or short-lived bursts before they saturate your link.
- **_Firewalls & ACLs (Access Control Lists):_** Configure strict rules at the edge of your network. Block unnecessary protocols (e.g., incoming ICMP/Ping) and restrict access only to essential ports (like 80/443).
- **_SYN Cookies:_** This is a critical defense against SYN Flood attacks. Instead of the server allocating memory (RAM) for every "half-open" connection, it sends a cryptographically generated "cookie" in the response. It only allocates resources after the connection is fully validated, preventing memory exhaustion.

### 2. Specialized DDoS Mitigation (The "Scrubbing" Layer)
Local hardware (on-premise firewalls) cannot stop massive Tbps-level volumetric attacks. You must leverage the scale of the cloud.
- **_DDoS Scrubbing Services:_** Providers like Cloudflare, Akamai, or AWS Shield act as a giant shield. They intercept your traffic globally, "scrub" or filter out the malicious packets, and forward only clean, legitimate traffic to your origin server.
- **_CDN (Content Delivery Network):_** By using a CDN, your actual Origin IP address is hidden behind the provider's IP. Attacks hit the CDN’s distributed edge nodes rather than your central server, making it nearly impossible for an attacker to pinpoint and "down" a single target.

### 3. Application Layer Defense (Layer 7)
Attackers often bypass the network layer to target expensive server-side operations, such as database queries or login pages.
- **_Rate Limiting:_** Set strict thresholds for how many requests a single IP can make per second. This prevents a bot from overwhelming your API or web server.
- **_WAF (Web Application Firewall):_** A WAF inspects the content of the traffic. It identifies bot-like behavior, malicious headers, or patterns (like a bot trying to scrape your site) and blocks them at the application level.
- **_CAPTCHA Challenges:_** When the system detects suspicious or high-velocity traffic, it can issue a CAPTCHA. This effectively stops automated bots while allowing human users to continue their session.


**_Tips: Never make "panic configurations" during a live attack. Changing firewall rules on the fly without a plan often leads to False Positives, where you accidentally block your own legitimate customers. Always have a pre-approved DDoS Playbook ready to execute._**
